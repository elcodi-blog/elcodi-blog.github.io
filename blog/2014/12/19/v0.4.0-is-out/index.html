0.4 Is Out!

Dec 19th, 2014

Hi all! It’s new minor version time in Elcodi! We did some work here, with some help from our community, which gets bigger every day. Proud! Thank you!

Here’s what we got for you:

  • #378 Pages
  • #382 Translator
  • #383 Configuration
  • #384 Fixed abstract extension

See the sections below for a description of each pull request.

Pages

With the new Page component you will be able to store content in database and serve it from any url you want, allowing you from static pages like “ToS” and “About Us” to a blog post. Manage your pages and add new content to your store.

Translator

The Translator will raise your entities to a whole new world of internazionalization with the ability to add translations for its fields. This component provides a solution fully decoupled from your model, so you don’t have to mess up with it. Good enough? Also, it’s integrated in the bamboo-backend so you can add new languages and translation with ease. Hola mundo!

Configuration

Many things should be configurable by your merchants, so we’ve come up with the Configuration component. It is a key-value manager so you can save and retrieve your settings easily: Shop name, main logo, tracking keysâ€¦ you name it. And I mean it! Just give the setting a name and you’ll be able to manage its value from the backend. Plain easy.

Fixed abstract extension

Also we made a full rework of AbstractExtension and AbstractConfiguration in the Core. See below for a detailed changelog to update to 0.4

  • Core/AbstractExtension:
    • The only mandatory method you need to implement is getAlias and return a unique string. As this is a well known practice, we expect the impact of this change to be minimal.
    • If you want to add custom configuration to your extension, implement getConfigurationInstance and return a Configuration object. If you extend it from AbstractConfiguration, remember to pass getAlias() as a parameter on construction.
  • Core/AbstractConfiguration:
    • Now the method getConfigTreeBuilder is defined for your convinience, so there’s no need for your configurations to provide it. You only have to implement setupTree, which receives the rootNode with its name from getAlias.

Other changes

In the “Don’t Repeat Yourself” side, all parameters for repository classes were removed and now you only have to supply the classname in your mapping files. Also, the old long service names are not used anymore, in favor of short alias.

Hope you find this changes useful! Let’s talk about it and share opinions on gitter.

Cheers!
